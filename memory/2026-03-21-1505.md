assistant: **Honestly? Yes, it could crash.**

**The problems:**

| Issue | What happens |
|-------|--------------|
| **Multiple processes** | Each agent runs its own process → no shared locks |
| **Zvec collection** | Single directory → multiple writers = potential corruption |
| **Markdown files** | Atomic writes ✅, but concurrent creates could have ID collisions |

**Scenario:**
```
cclow creates C1 → writes to index
finan creates C1 → overwrites index (ID collision)
```

**Current state:**
- **Single agent:** Works fine ✅
- **Multiple agents, same process:** Works (async, but no true parallelism)
- **Multiple agents, different processes:** ❌ **Danger zone**

**To make it multi-agent safe:**

| Solution | Effort |
|----------|--------|
| **Single API server** (all agents call same API) | Low — already have `vtic serve` |
| **File locking** (flock on operations) | Medium |
| **Database backend** (replace Zvec with SQLite/Postgres) | High |
| **Message queue** (Redis pub/sub) | Medium |

**Quickest fix:** All agents should use **`vtic serve` API**, not direct file access:

```bash
# Run ONE server
vtic serve --port 8900

# All agents call API
curl -X POST http://localhost:8900/tickets -d '{"title": "..."}'
```
